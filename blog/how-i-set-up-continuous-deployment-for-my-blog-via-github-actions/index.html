How I set up continuous deployment for my blog via Github Actions

July 14. 2021
  • hugo
  • github

I thought I’d christen this blog in the traditional way, by talking about how I set it up in the first place. I’m using Hugo to build this site, and at some point I’ll go through the set up I used, but this post is specifically about how I use Github Actions to deploy this blog from a (free, private) source repo to a (public) target repo which uses Github Pages.

Why would you do it this way?

You can create and host a Github Pages website out of a single public repo. However, this means exposing all your working to the world: all your drafts, the seven commits in a row where you couldn’t make your mind up about styling, and even the old deleted posts still lurking in your git history. If, like me, you’d rather keep that stuff separate and private, it makes sense to separate your website’s source code from the public-facing repository where it’s hosted.

Back in the olden days, using two repos this way meant you’d need to keep them both on your local computer, and fiddle around with custom bash scripts to move build output between one repo and the other, or (even better) learn how to use git submodules. Nowadays, everybody loves continuous integration, and you can use Github Actions to continuously, automatically deploy your static site.

The repos

First you need to have a target repo set up. By default, Github will try to publish a repo if it is named <user>.github.io or <organization>.github.io, and if that repo is public. (You can also publish from private repos if you have a paid plan, which I don’t.) We won’t go into how to set one of these up but you can follow Github’s guide here. For me this gives a repo called stevde/stevde.github.io. You won’t need to clone this repo.

Next you’ll need a source repo. I’m using Hugo here, but you can use anything you like, as long as it generates a static website in a given folder. Again we won’t go into how to set up Hugo here but you can try their Quick Start guide if you’re curious. Once everything’s built, I have a repo layout that looks a little like this:

    # licenses and documentation
├── LICENSE
├── README.md
    # main hugo config file
├── config.toml
    # hugo archetypes and content dirs
├── archetypes/
├── content/
    # files for controlling hugo's layout and styling
├── layouts/
├── static/
    # the main output directory 
    # (finished website goes in here)
└── public/

All of this code is committed to a private Github repo (for me, stevde/blog-source).

Configuring Github workflows

Now that we have our repos set up, we need to find a way to take the files from our public/ directory in our source repo and put them in the target repo (the target repo will only contain the files from our public/ directory). This is where Github Actions come in. Actions is Github’s native CI offering, and it allows you to specify a set of steps to be run automatically after a given trigger. You can set up your own custom actions or use ready-made actions from the marketplace.

We have a very specific process in mind: After a successful push to origin/master, we want to:

  1. Start an Ubuntu instance.
  2. Install Hugo.
  3. Build our website using Hugo, putting the output in public/.
  4. Deploy the contents of public/ to our public repository which has github pages set up.

This section is heavily based on the official Hugo setup guide for Github Pages and peaceiris' very handy documentation for their github actions repos. I’d recommend reading those first, they’ll get you 90% of the way there. What I add below is mostly notes for my own reference and other things which took me a while to figure out.

To get this going, create the following file structure in the root of your source repo:

.github
└── workflows
    └── gh-pages.yml

Then put the following config into gh-pages.yml:

# name of the workflow
name: github pages

# triggered on pushes to master, 
# or any pull request
on:
  push:
    branches:
      - master
  pull_request:

# jobs to be run on trigger
jobs:
  deploy:
    # specify working environment
    runs-on: ubuntu-20.04
    steps:
      # checkout repo
      - uses: actions/checkout@v2

      # install Hugo
      - name: Setup Hugo
        uses: peaceiris/actions-hugo@v2
        with:
          hugo-version: "latest"

      # build site
      - name: Build
        run: hugo --minify

      # if on master branch, deploy
      - name: Deploy
        uses: peaceiris/actions-gh-pages@v3
        if: github.ref == 'refs/heads/master'
        with:
          deploy_key: ${{ secrets.DEPLOY_KEY }}
          external_repository: stevde/stevde.github.io
          publish_branch: master
          publish_dir: ./public

The first part is simple enough, it specifies the name of the workflow and the triggers we want to look out for:

# name of the workflow
name: github pages

# triggered on pushes to master, 
# or any pull request
on:
  push:
    branches:
      - master
  pull_request:

Note that this our workflow will be triggered after any push to master or after any pull request is created (regardless of the branch name). You can find more info on triggers here.

Next we begin to specify the workflow. The first few steps in the workflow are quite self-explanatory:

# jobs to be run on trigger
jobs:
  deploy:
    # specify working environment
    runs-on: ubuntu-20.04
    steps:
      # checkout repo
      - uses: actions/checkout@v2

      # install Hugo
      - name: Setup Hugo
        uses: peaceiris/actions-hugo@v2
        with:
          hugo-version: "latest"

      # build site
      - name: Build
        run: hugo --minify

Here we create a new Ubuntu container, checkout the current version of our source repo, install Hugo and build the site. Note that, because the public/ directory is always newly created after a push, you don’t need to have it on your local or keep it in git history. I add public/ to my repo’s .gitignore.

Finally, we get to the magic final step:

      # if on master branch, deploy
      - name: Deploy
        uses: peaceiris/actions-gh-pages@v3
        if: github.ref == 'refs/heads/master'
        with:
          deploy_key: ${{ secrets.DEPLOY_KEY }}
          external_repository: stevde/stevde.github.io
          publish_branch: master
          publish_dir: ./public

This one’s a little more involved, so let’s go through it line by line. First we specify the name and the action we’ll be using (again, it’s peaceiris' super-useful action: peaceiris/actions-gh-pages):

      - name: Deploy
        uses: peaceiris/actions-gh-pages@v3

Next we specify that this step will only be run if we’re on the master branch of our repo:

        if: github.ref == 'refs/heads/master'

This is important! In general, we want as much of our workflow as possible to run when a PR is opened (because then we get to check that the changes in the PR still build). But we definitely don’t want to deploy PRs before they’re accepted to master. This condition essentially filters only for triggers coming directly from master. See more examples in the actions syntax documentation.

Finally, we have a with:, which provides the environment variables used by actions-gh-pages:

        with:
          external_repository: stevde/stevde.github.io
          publish_branch: master
          publish_dir: ./public
          deploy_key: ${{ secrets.DEPLOY_KEY }}

First we specify the target repo (my public repo which has Github Pages setup):

          external_repository: stevde/stevde.github.io

Next we set the branch and directory in the source repo where we’ll take our content from:

          publish_branch: master
          publish_dir: ./public

Finally, we set a deploy_key, which the action will use to gain access to the target repo and add new files.

          deploy_key: ${{ secrets.DEPLOY_KEY }}

This last line was a little difficult to figure out based on the docs, so I added a separate section on it below.

Authenticating the workflows

So what exactly is ${{ secrets.DEPLOY_KEY }}?

If you’ve ever ssh’d into anything before, you know that you need a two-part key in order to get authenticated: a private key that sits on your computer and a public key which you can give to other machines that you’d like to access. When the final step of our workflow runs, we are, in essence, sshing from the ubuntu container in our source repo into our target repo. So we need to set up the private part of our ssh key in our source repo, and give the public part to our target repo. Let’s walk through this process.

First we need to generate our keys. Let’s do this, following Github’s recommendations for key generation:

ssh-keygen -t ed25519 -C <your-github-email> -f ~/.ssh/github-pages-key

This will create two new files in ~/.ssh/:

  • github-pages-key.pub : the public part of your key
  • github-pages-key: the private part of your key

Following the metaphor of sshing from our source repo to our target repo, we first need to add our public key to our target repo. Go to your target repo, click Settings then Deploy keys, then Add new deploy key. Here you can paste in the contents of github-pages-key.pub, give it a name, be sure to tick Allow write access, and click Add key.

After that, we can add our private key to our source repo. In general, you should be pretty wary about putting private keys anywhere, but since this particular key is only going to get used for this deployment, it’s safe to give it to Github. Go to your source repo, click Settings then Secrets, and click New repository secret. Give it a name (here we use DEPLOY_KEY), and paste the contents of github-pages-key.

Finally (finally!) we are able to access the value of this secret by using the Github-specific syntax ${{ secrets.DEPLOY_KEY }}.

End result

By this point, you should have a working continuous deployment pipeline where your static site is redeployed any time you push new changes to master. You can test it out by making a change, committing, and pushing. You should be able to see your workflow running under the Actions tab of your source repo, and you should be able to see the new commits automatically generated in your target repo. Good luck and happy deploying!
